# AquaGuide 云同步方案评估（不实施）

> 状态：方案评估。本文不是数据契约，不包含可执行迁移或建表 SQL；在用户确认 `CONTRACT.md`、字段、RLS 和迁移策略前，不接入 Supabase 业务数据。

## 1. 建议同步的实体

| 实体 | 当前来源 | 云端建议边界 | 暂不同步 |
|---|---|---|---|
| 用户鱼缸 | `aquariums` / 主状态 | 每个鱼缸一行，生物、设备和环境拆为受控子表或 JSON 契约 | 3D 临时渲染状态 |
| 巡检与诊断 | `diagnosisRecords` | 结构化答案、规则结论、行动、观察项 | AI 原始回复、自由描述原文默认不上云 |
| 收藏 | 现有物种与养护收藏键 | 用户、内容类型、内容 ID、收藏时间 | 重复缓存的文章正文 |
| 生命纪念 | `deceasedRecords` | 物种、日期、复盘原因、关联鱼缸 | 派生成就状态 |
| 养护记录 | 换水、喂食、观察和养护清单 | 统一事件模型，保留发生时间与鱼缸 | 纯 UI 展开状态 |
| 成就 | 由已有记录计算 | 不建独立业务表，继续实时派生 | 积分、排行、手动领取 |

## 2. 表结构方向

- `profiles`：只保存账号展示与偏好，不复制认证凭据。
- `aquariums`：用户归属、名称、水体、尺寸、目标温度与更新时间。
- `aquarium_species`：鱼缸、物种 ID、数量、入缸日期。
- `aquarium_equipment`：鱼缸与过滤、加热、增氧、灯光配置。
- `diagnosis_records`：鱼缸、自然日、问题类型、结构化答案、确定性结论和安全行动。
- `favorites`：用户、`species | care` 类型、内容 ID，建立唯一约束避免重复。
- `memorial_records`：用户、鱼缸、物种、日期与原因。
- `care_events`：换水、喂食、观察、提醒和清单事件。

正式契约必须统一 ID、时间、可空字段、删除策略和浏览器 camelCase ↔ 数据库 snake_case 转换；本阶段不锁定字段。

## 3. RLS 原则

1. 每张用户业务表都必须能直接或经鱼缸外键追溯到 `auth.uid()`。
2. 读、增、改、删分别建策略，不能只依赖前端筛选。
3. 物种库和养护文章为公共只读；用户不能覆盖官方内容。
4. 服务端 AI 只读取完成任务所需的最小鱼缸快照，不持有跨用户查询权限。
5. 删除账号时使用明确的级联或异步清理策略，并提供导出窗口。

## 4. 本地迁移流程

1. 登录后先生成只读迁移预览：实体数量、异常记录、重复项和无法识别物种。
2. 用户确认后上传一个带 `migration_id` 的批次；服务端按唯一键幂等写入。
3. 上传成功后回读数量与摘要，比对无误才标记迁移完成。
4. 本地数据至少保留 30 天或直到用户主动清除；失败不能破坏原数据。
5. 旧键继续兼容读取一个版本周期，不能在首次登录时静默删除。

## 5. 冲突合并

- 鱼缸配置：字段级 `updated_at`，冲突时展示本机/云端摘要，由用户选择。
- 数量与记录：使用事件或记录 ID 幂等合并，禁止简单相加。
- 收藏：集合并集；取消收藏使用删除时间避免旧设备复活。
- 同日巡检：仍保持“每缸每天一条”，以后更新时间较新的结构化记录为主，并保留冲突审计。
- 生命纪念：按记录 ID 合并，不依据物种和日期自动去重。

## 6. 备份、恢复与隐私

- 上云前继续保留当前 JSON 导入导出；云端提供按用户导出与删除。
- 备份需覆盖误删恢复目标、保存周期和恢复演练，不把“有数据库”当作备份。
- 自由描述可能包含个人信息，默认只在本次 AI 请求中使用，不持久化原文。
- 产品需说明同步内容、保存地区、删除方式和 AI 数据流；未登录用户继续完全本地可用。
